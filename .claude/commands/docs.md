description: コードドキュメントの生成
---

あなたはClaudeCodeです。現在のコードを解析し、包括的なドキュメントを**日本語で**生成してください。
すべてのコメント、説明、ドキュメントは日本語で記述してください。

## 実行手順

1. **対象コードの読み取り**
   - 指定されたファイルまたはディレクトリの内容を確認する
   - 依存関係のあるファイルも必要に応じて読み取る

2. **ドキュメント内容の決定**
   コードの種類に応じて適切なドキュメントを生成する：

   **関数・メソッドの場合**
   - 関数の目的と概要
   - パラメータの説明（型、デフォルト値、制約）
   - 戻り値の説明（型、取りうる値）
   - 例外・エラーの説明
   - 使用例
   - 注意事項

   **クラスの場合**
   - クラスの目的と責務
   - プロパティの説明
   - メソッドの説明
   - 継承関係
   - 使用例
   - 設計上の考慮事項

   **モジュール・パッケージの場合**
   - モジュールの目的
   - 提供する機能の概要
   - 主要なエクスポート
   - 依存関係
   - インストール方法
   - 基本的な使い方
   - API リファレンス

   **プロジェクト全体の場合**
   - プロジェクトの概要
   - 主な機能
   - アーキテクチャ
   - ディレクトリ構成
   - セットアップ手順
   - 使用方法
   - 開発ガイド
   - コントリビューションガイド

3. **ドキュメント形式の選択**
   プロジェクトの慣習に従って適切な形式を選択：
   - JSDoc（JavaScript/TypeScript）
   - docstring（Python）
   - RDoc（Ruby）
   - JavaDoc（Java）
   - XML Documentation Comments（C#）
   - Markdown（README、ガイド）

4. **ドキュメントの生成**
   - 選択した形式でドキュメントを作成する
   - コード例は動作確認済みのものを使用する
   - 専門用語には説明を付ける
   - 図表が必要な場合はMermaid記法などを使用する

5. **ドキュメントの配置**
   以下のいずれかの方法でドキュメントを配置：
   - インラインドキュメント：コード内にコメントとして追加
   - 外部ドキュメント：README.md、docs/ディレクトリなどに配置
   - 両方：インラインドキュメントと外部ドキュメントの両方を生成

## ドキュメント品質の基準

生成するドキュメントは以下の基準を満たすこと：

- **正確性**: コードの実際の動作と一致していること
- **完全性**: 必要な情報がすべて含まれていること
- **明確性**: 専門知識のないユーザーでも理解できること
- **簡潔性**: 冗長でなく、要点を押さえていること
- **一貫性**: プロジェクト内の他のドキュメントと表記が統一されていること
- **実用性**: 実際の使用に役立つ例が含まれていること

## オプション引数

ユーザーが追加の引数を指定した場合：
- `--inline`: インラインドキュメントのみ生成
- `--external`: 外部ドキュメントファイルのみ生成
- `--format=<形式>`: ドキュメント形式を明示的に指定
- `--lang=<言語>`: ドキュメントの言語を指定（デフォルト: 日本語）
- `--examples`: より多くの使用例を含める

## 注意事項

- 既存のドキュメントがある場合は、それを尊重しつつ不足部分を補完する
- コードを変更しない（ドキュメントのみ追加・更新）
- 自動生成であることを示すコメントは付けない
- プロジェクトのドキュメント規約に従う
